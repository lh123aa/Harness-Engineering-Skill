## 1. 系统架构

```
┌──────────────────────────────────────────────────────────────────┐
│                    自我迭代系统 (Self-Iteration System)          │
│                                                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐     │
│  │ 阶段 1   │ → │ 阶段 2   │ → │ 阶段 3   │ → │ 阶段 4   │     │
│  │ 质量扫描 │   │ 问题分类 │   │ 修复执行 │   │ 回归验证 │     │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘     │
│       │              │              │              │             │
│       ▼              ▼              ▼              ▼             │
│  20 项检查点    P0/P1/P2      修复策略库     回归验证清单         │
│       │                                              │           │
│       └────────────────── 循环 ──────────────────────┘           │
│                          ↓ 退出条件                                │
│                    问题计数 == 0                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.1 循环控制器

```
function SelfIterationLoop(targetFile):
  while true:
    issues = QualityScan(targetFile)        # 阶段 1
    if issues.count == 0: break             # 归零退出
    classified = Classify(issues)           # 阶段 2
    for issue in classified:
      Fix(targetFile, issue)                # 阶段 3
    Verify(targetFile)                      # 阶段 4
    BumpVersion(targetFile)                 # 版本递增
  return "✅ 系统问题为零，迭代终止"
```

### 1.2 三层部署模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **嵌入模式** | 将质量检查点嵌入 Skill 的 System Prompt 中，AI 在输出前自我检查 | AI Chat、IDE 内嵌 |
| **审计模式** | 用独立的 Skill Auditor 扫描目标 Skill 文件，输出问题报告 | 文档审查、CI 质量门禁 |
| **自动化模式** | 脚本驱动完整循环（扫描→分类→修复→验证→循环） | 持续集成、批量维护 |

## 2. 质量检查点（20 项）

### 2.1 格式层（6 项）

| ID | 检查点 | 检查方法 | 修复策略 |
|----|--------|----------|----------|
| F-01 | **代码块平衡** | 统计 ` ``` ` 出现次数，必须为偶数 | 补充缺失的闭合/开启 fence |
| F-02 | **无自评分** | 搜索 "评分.*10" "10.*分" "满分" 等模式 | 移除自评分行 |
| F-03 | **无多余代码块** | 搜索 ````Plain Text` 等孤立块 | 删除或闭合 |
| F-04 | **锚点可导航** | 目录中的 `(#...)` 链接在文档中存在对应 `##` 标题 | 修正锚点名或标题文本 |
| F-05 | **无断链引用** | Prompt 内引用外部章节（如"参见 §X"）| 改为自包含描述 |
| F-06 | **嵌套代码块缩进一致** | 代码块内嵌代码块时缩进 >= 4 空格 | 统一缩进 |

### 2.2 语义层（7 项）

| ID | 检查点 | 检查方法 | 修复策略 |
|----|--------|----------|----------|
| S-01 | **原则完整度** | 核心原则列表是否连续编号、无跳号 | 补全缺失原则或重编号 |
| S-02 | **流程终态定义** | 每个分支（轻量/标准/完整）都有明确的终点 | 补充缺失的流程出口指令 |
| S-03 | **提前终止定义** | 是否有「先到这里」「后续再说」的处理逻辑 | 补充提前终止协议 |
| S-04 | **模糊响应处理** | 是否有对「不知道」「随便」的处理规则 | 补充模糊响应原则 |
| S-05 | **输入校验机制** | 是否有对不合理输入（过多 P0、矛盾描述）的 challenge 规则 | 补充输入校验原则 |
| S-06 | **术语一致性** | 同一概念在全文中用词一致（如"可选"vs"非必需"）| 统一术语 |
| S-07 | **无承诺越界** | 没有超出 Skill 能力边界的承诺（如"自动分配多 Agent"）| 降低为诚实说明 |

### 2.3 逻辑层（4 项）

| ID | 检查点 | 检查方法 | 修复策略 |
|----|--------|----------|----------|
| L-01 | **模式间不冲突** | 轻量/标准/完整三个路径的指令不互相矛盾 | 消除矛盾指令 |
| L-02 | **原则与流程不冲突** | 原则说「X」但流程做「非X」| 统一原则与流程 |
| L-03 | **版本号一致** | 文档头、尾、版本历史中的版本号一致 | 统一改为最大版本号 |
| L-04 | **出口可达** | 每个路径最终都能到达文档生成阶段 | 补充缺失的路径连接 |

### 2.4 稳健层（3 项）

| ID | 检查点 | 检查方法 | 修复策略 |
|----|--------|----------|----------|
| R-01 | **无未定义术语** | 文档引入的概念都有定义（如"阶段性草案文档"）| 补充术语定义 |
| R-02 | **空值处理** | 用户不回答时 AI 的行为有定义 | 补充默认值或追问策略 |
| R-03 | **中断恢复** | 对话中断后能否基于历史继续 | 补充状态记录指令 |

## 3. 自我迭代协议（System Prompt 嵌入版）

以下内容可嵌入任意 Skill 的 System Prompt 末尾，使其具备自我评估能力：

```markdown
# 自我迭代协议（嵌入模块）

## 质量自查（输出前必执行）
每次生成文档前，执行以下 6 项快速自查：

```
□ 1. 输出是否遵循了模板结构？（对照输出格式规范）
□ 2. 是否遗漏了用户指定的 P0 功能？
□ 3. 当前处于哪个阶段？是否有未完成的前置步骤？
□ 4. 用户是否有未回答的问题需要再次询问？
□ 5. 本次输出是否引入了前后矛盾？（对照之前已确认的信息）
□ 6. 是否有超出 Skill 能力边界的承诺？
```

任一项为「否」则修正后再输出。

## 迭代反馈协议
如果在运行过程中发现 Skill 自身的指令存在以下问题，**记录到对话末尾的改进日志**：

| 问题类型 | 记录格式 |
|----------|----------|
| 指令模糊 | `[改进建议] 第X条原则「……」表述模糊，建议改为「……」` |
| 流程缺失 | `[改进建议] 场景「……」的流程未定义，建议补充` |
| 逻辑矛盾 | `[改进建议] 原则X说「……」但流程Y要求「……」矛盾` |
| 边界越权 | `[改进建议] 「……」超出 Skill 能力边界，建议修改为「……」` |
| 术语不一 | `[改进建议] 「……」和「……」指向同一概念，建议统一` |

## 版本自检
每次迭代后，检查：
- 文件头版本号与版本历史最后一条一致
- 版本历史中最新条目的日期为当天
- 所有版本号引用（JSON 模板等）同步更新
```

## 4. 自动化循环引擎

### 4.1 PowerShell 循环脚本（自迭代驱动器）

以下脚本可驱动任意 Skill 文档的自我迭代循环：

```powershell
# Self-Iteration Engine v1.0
# 用法: .\SelfIterate.ps1 -TargetFile "path\to\skill.md"

param(
    [Parameter(Mandatory=$true)]
    [string]$TargetFile,
    [int]$MaxIterations = 10
)

$iteration = 0
$issueLog = @()

while ($iteration -lt $MaxIterations) {
    $iteration++
    Write-Host "`n=== 迭代 $iteration ===" -ForegroundColor Cyan
    
    # 阶段 1: 质量扫描
    Write-Host "[阶段 1] 质量扫描..." -ForegroundColor Yellow
    $issues = @()
    
    # F-01: 代码块平衡
    $content = Get-Content $TargetFile -Raw
    $fenceMatches = [regex]::Matches($content, '^```', 'Multiline')
    if ($fenceMatches.Count % 2 -ne 0) {
        $issues += [PSCustomObject]@{ Id="F-01"; Severity="P0"; Desc="代码块不平衡 ($($fenceMatches.Count) 个 fence)" }
    }
    
    # F-02: 自评分
    if ($content -match '评分.*10[\s\.]') {
        $issues += [PSCustomObject]@{ Id="F-02"; Severity="P0"; Desc="存在自评分" }
    }
    
    # F-03: 多余 Plain Text 块
    if ($content -match '```Plain Text') {
        $issues += [PSCustomObject]@{ Id="F-03"; Severity="P0"; Desc="存在残余 Plain Text 代码块" }
    }
    
    # L-03: 版本号一致
    $headerMatch = [regex]::Match($content, '(?<=版本[：:]\s*)v?(\d+\.\d+)')
    $footerMatch = [regex]::Match($content, '(?<=文档版本[：:]\s*)v?(\d+\.\d+)')
    if ($headerMatch.Success -and $footerMatch.Success) {
        if ($headerMatch.Groups[1].Value -ne $footerMatch.Groups[1].Value) {
            $issues += [PSCustomObject]@{ Id="L-03"; Severity="P1"; Desc="头尾版本号不一致: $($headerMatch.Groups[1].Value) vs $($footerMatch.Groups[1].Value)" }
        }
    }
    
    # F-05: 外部引用
    if ($content -match '参见[^。]*§[^。]*。' -or $content -match 'refer to section') {
        $issues += [PSCustomObject]@{ Id="F-05"; Severity="P1"; Desc="Prompt 内存在外部章节引用" }
    }
    
    # R-01: 未定义术语
    $knownTerms = @('阶段性草案文档', 'P0', 'P1', 'P2', 'AGENTS\.md', 'MVP')
    $allTerms = [regex]::Matches($content, '[「」""][^「」""]{2,20}[「」""]') | ForEach-Object { $_.Value }
    # 简化检查：搜索「xxx」模式并验证是否在前文有定义
    # (此检查在纯脚本中较复杂，建议在 AI 审计模式中执行)
    
    # 更多检查点可按需扩展...
    
    # 阶段 2: 问题分类
    if ($issues.Count -eq 0) {
        Write-Host "[阶段 2] 零问题，迭代终止" -ForegroundColor Green
        break
    }
    
    Write-Host "[阶段 2] 发现 $($issues.Count) 个问题:" -ForegroundColor Yellow
    $issues | Group-Object Severity | ForEach-Object {
        Write-Host "  $($_.Name): $($_.Count) 项" -ForegroundColor $(
            if ($_.Name -eq 'P0') { 'Red' } elseif ($_.Name -eq 'P1') { 'Yellow' } else { 'Gray' }
        )
    }
    
    # 阶段 3: 修复执行（自动修复的项）
    Write-Host "[阶段 3] 执行自动修复..." -ForegroundColor Yellow
    # 修复策略在脚本中实现在线替换
    # 当前仅做日志记录，复杂修复需 AI 辅助
    
    # 阶段 4: 版本递增
    Write-Host "[阶段 4] 递增版本号..." -ForegroundColor Yellow
    
    # 记录本次迭代
    $issueLog += [PSCustomObject]@{
        Iteration = $iteration
        IssueCount = $issues.Count
        Issues = ($issues | ForEach-Object { "$($_.Id)($($_.Severity))" }) -join ', '
    }
    
    Start-Sleep -Milliseconds 500  # 防止 IO 冲突
}

# 输出迭代报告
Write-Host "`n=== 迭代报告 ===" -ForegroundColor Cyan
$issueLog | Format-Table -AutoSize

if ($issues.Count -eq 0) {
    Write-Host "✅ 系统问题为零，退出循环" -ForegroundColor Green
} else {
    Write-Host "⚠️ 达到最大迭代次数 ($MaxIterations)，仍有 $($issues.Count) 个问题未解决" -ForegroundColor Yellow
}
```

### 4.2 关键设计原则

| 原则 | 说明 |
|------|------|
| **幂等性** | 重复运行同一检查点应得到相同结果（无状态残留） |
| **最小修改** | 每次修复只改必要行，不引入无关变更 |
| **版本追溯** | 每次迭代递增版本号，版本历史记录变更内容 |
| **终止保障** | 设置最大迭代次数（默认 10 次），防止无限循环 |

## 5. 修复策略库

### 5.1 自动修复（脚本可执行）

| 问题 | 修复规则 |
|------|----------|
| 版本号不一致 | 将所有版本号统一为最新版本 |
| 自评分行 | 删除包含「评分.*10」的行 |
| 多余 Plain Text | 删除包含 ````Plain Text` 的行 |
| 锚点错位 | 将目录中锚点名修改为对应标题的 GitHub 风格 slug |

### 5.2 半自动修复（需 AI 判定）

| 问题 | 修复策略 | AI 判断点 |
|------|----------|-----------|
| 流程缺失 | 根据上下文补充缺失的流程指令 | 补充内容是否与现有逻辑一致 |
| 术语不一致 | 统一为首次出现或使用频率最高的术语 | 哪个术语更准确 |
| 原则-流程矛盾 | 调整原则或流程使其一致 | 哪个更符合设计意图 |
| 边界越权 | 降低承诺等级或补充能力边界说明 | 是否真实超出能力范围 |

### 5.3 修复执行协议

```
每条修复必须遵循：
1. 读取目标行前后 5 行上下文（防误改）
2. 执行最小差异修改（只改问题行）
3. 修改后立即验证（再次运行对应检查点）
4. 验证通过则继续，失败则回退并记录
```

## 6. 归零条件与退出机制

### 6.1 归零判定

系统在以下条件**全部满足**时判定为「零问题」：

```
□ F-01 代码块平衡: 通过
□ F-02 无自评分: 通过
□ F-03 无多余代码块: 通过
□ F-04 锚点可导航: 通过
□ F-05 无断链引用: 通过
□ F-06 嵌套缩进一致: 通过
□ S-01 原则完整度: 通过
□ S-02 流程终态定义: 通过
□ S-03 提前终止定义: 通过
□ S-04 模糊响应处理: 通过
□ S-05 输入校验机制: 通过
□ S-06 术语一致性: 通过
□ S-07 无承诺越界: 通过
□ L-01 模式间不冲突: 通过
□ L-02 原则与流程不冲突: 通过
□ L-03 版本号一致: 通过
□ L-04 出口可达: 通过
□ R-01 无未定义术语: 通过
□ R-02 空值处理: 通过
□ R-03 中断恢复: 通过
```

### 6.2 退出条件矩阵

| 条件 | 操作 |
|------|------|
| 全部 20 项通过 | ✅ **正常退出** — 输出零问题报告 |
| 连续 3 次迭代无变化 | ⚠️ **强制退出** — 标记为"卡死"，输出当前问题清单 |
| 达到最大迭代次数 | ⚠️ **超限退出** — 输出未解决问题清单 |
| 修复引入新问题 | 🔄 **回退并退出** — 撤销最后一次修复，输出报告 |
